# Artone v3 — Apple 設計原則

全ての設計判断はこの文書を基準とする。
機能追加の提案時に「この原則のどれに合致するか」を答えられなければ却下。

---

## 1. 1000 の No

新機能を追加する前に問う:

- **この機能がなくてもプロダクトは成立するか?** → Yes なら不要
- **この機能は 90% のユーザーが使うか?** → No なら Pro tier に隔離
- **この機能は既存機能と重複していないか?** → Yes なら統合
- **この機能は説明なしで使えるか?** → No なら設計を見直す

実行済み: streaming / cloud / mobile / batch は Pro tier に隔離。Essential tier は 7 コマンドのみ。

## 2. 段階的開示 (Progressive Disclosure)

| Tier | 表示条件 | 機能例 |
|---|---|---|
| Essential | 常時表示 | カット、再生、エクスポート |
| Standard | プロジェクト保存後 | カラー、字幕、キーフレーム |
| Pro | 設定で有効化 | マルチカム、OTIO、ライブ配信 |

原則: ユーザーの成長に合わせて UI が成長する。初回起動時に 30 個のボタンを見せない。

## 3. 主役を決める

画面の中で「最も重要な 1 つ」を決め、それに 60% 以上の面積を与える。

Artone の主役: **タイムライン**

- タイムラインは画面の 60%+ を占める
- プレビューは 40% 以下
- サイドバーは折りたたみ可能
- ツールバーは最小限

## 4. デザインシステムが法律

`app/design-system.ts` が全 UI の唯一の真実。

禁止:
- 色の直書き (`#FF0000` ではなく `color.destructive`)
- フォントサイズの直指定 (`14px` ではなく `ds.text('body')`)
- スペーシングの直指定 (`16px` ではなく `space[4]`)
- 独自テーマ定数 (`const T = {...}` は禁止)

## 5. 1 つの意味に 1 つの色

| 役割 | 色 | 使用箇所 |
|---|---|---|
| ブランド | #00C4CC | ロゴ、アクセント、フォーカスリング |
| 操作 | #3B82F6 | クリック可能なアクション |
| 成功 | #10B981 | 完了、正常 |
| 注意 | #F59E0B | 未保存、警告 |
| 危険 | #EF4444 | 削除、エラー |
| プレイヘッド | #FF3B30 | タイムライン上の再生位置のみ |

同じ色を 2 つの意味で使わない。

## 6. アニメーションは意味を持つ

| 動き | 用途 | 時間 |
|---|---|---|
| spring | 要素の登場 (弾み) | 350ms |
| snappy | 状態変化 | 120ms |
| slide | パネル開閉 | 250ms |
| fade | 背景変化 | 200ms |

無意味なアニメーションは削除。ユーザーの注意を操作に向けるのが目的。

## 7. エラーを見せない

- 自動保存は UI に表示しない (失敗時のみ通知)
- プロキシ生成は裏で自動実行
- コーデック判定は自動
- エクスポート設定は 1 ボタン (カスタムは展開)

原則: ユーザーが技術的な判断を迫られる場面をゼロにする。

## 8. Cmd+K が全て

全てのコマンド・エフェクト・設定は Cmd+K (Command Palette) で到達可能。

- メニューバーは不要
- コンテキストメニューは補助
- ツールバーは最頻コマンドのみ
- それ以外は Cmd+K

これにより:
- ショートカットを覚えなくても使える
- 機能発見性が高い (検索で見つかる)
- UI がシンプルに保たれる

## 9. 触って気持ちいい

- ホバー: 即座に反応 (120ms)
- クリック: フィードバック (haptic 的な scale/opacity)
- ドラッグ: 物理演算的な追従
- スクロール: 慣性付き

Apple 製品の「手で触れている感覚」を Web で再現する。

## 10. 削る勇気

四半期ごとに「最も使われていない機能」を特定し、削除を検討する。

判断基準:
- 使用率 < 5% かつ代替手段あり → 削除候補
- 使用率 < 1% → 即削除
- コードの保守コスト > ユーザー価値 → 削除

追加は慎重に、削除は積極的に。
